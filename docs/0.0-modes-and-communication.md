# 7 Mode & Cách các agent giao tiếp

Doc này định nghĩa **7 chế độ tương tác** giữa user và hệ thống, cách **conversation** chia tách, **state** flow giữa các turn, và **quy tắc giao tiếp** giữa agent.

## 7 mode

| Mode | Entry | Routing |
|---|---|---|
| **0 · Free chat** | User gõ vào ô free chat (không click group) | Supervisor classify → route 1 trong 6 Manager |
| **1 · Locked nhóm A** | User click card "Khám phá bản thân" | Bypass Supervisor → vào thẳng Manager A |
| **2 · Locked nhóm B** | User click card "Tư vấn nghề nghiệp" | Bypass Supervisor → Manager B |
| **3 · Locked nhóm C** | User click card "Tìm việc làm" | Bypass Supervisor → Manager C |
| **4 · Locked nhóm D** | User click card "Phân tích thị trường" | Bypass Supervisor → Manager D |
| **5 · Locked nhóm E** | User click card "Tối ưu hồ sơ" | Bypass Supervisor → Manager E |
| **6 · Locked nhóm F** | User click card "Phỏng vấn & đàm phán" | Bypass Supervisor → Manager F |

## Conversation rule

| Mode | Rule |
|---|---|
| **Mode 0** | 1 conversation duy nhất cho free chat. Supervisor có thể route tới các Manager khác nhau qua các turn — vẫn cùng `conversation_id`. |
| **Mode 1-6** | Mỗi mode = 1 conversation độc lập. **Switch mode → new conversation.** State per-mode không carry. |

### Hệ quả

UX giống **ChatGPT** — user có **danh sách conversation cũ trong sidebar**, tại bất kỳ thời điểm chỉ đang **active 1 conversation**.

```
User vào app → sidebar:
  • [Mode 0] "Tư vấn tìm việc DA — 2 ngày trước"
  • [Mode 2] "Discovery MBTI — tuần trước"
  • [Mode 3] "Tìm việc AI Engineer — hôm qua"
  ...

Hành vi:
  - Click thread cũ → resume conversation (cùng conversation_id cũ).
  - Click "New chat" → tạo conversation mới Mode 0 (free chat).
  - Click card group X từ UI → tạo conversation mới Mode X (locked).
```

- **Memory + Profile carry across mọi conversation** (user-scoped, không conversation-scoped) — info user nhập ở conversation A vẫn dùng được ở conversation B sau khi Profile/Memory cập nhật.
- State per-mode (vd Job Search `last_search.filter`) **chỉ valid trong 1 conversation**. Switch sang conversation khác → state không carry.

## Memory + Profile chạy ở mọi mode

```
User input ──┬─► [routing: Supervisor hoặc Manager X locked] → sub-agent → response
             ├─► Agent 2 · Memory     (song song, mọi mode)
             └─► Agent 3 · Profile    (song song, mọi mode)
```

- Hoạt động **identical** ở cả 7 mode.
- Nhận mọi user message (bất kể routing đi đâu) → extract & update.
- User-scoped — không quan tâm `conversation_id`, không quan tâm mode.
- Mới có hiệu lực ở turn **N+1** (trade-off accept để không block luồng chính).

## Compound question — xử lý theo mode

Khi 1 message chứa intent của nhiều group:

### Mode 0 (Free chat) — Supervisor pick + flag

```
User: "Tìm việc DA HN 20tr + luyện phỏng vấn"
  ↓ Supervisor classify
  ↓ Detect compound (≥2 intent group)
  ↓ Pick best fit → Manager C (phần lớn nhất)
  ↓ Append note vào response cho Manager C:
    "Câu của bạn có phần về luyện phỏng vấn (nhóm F).
     Mình xử lý phần tìm việc trước. Phần luyện phỏng vấn,
     bạn hỏi lại sau hoặc chuyển sang nhóm F nhé."
```

### Mode 1-6 (Locked) — Manager xử lý in-scope + flag off-scope

```
User trong Mode B gõ:
  "Lộ trình học DA + lương DA bao nhiêu?"
  ↓ Manager B nhận
  ↓ Detect compound, tách:
    - In-scope (B): lộ trình học DA → call Learning Path
    - Off-scope (D): lương DA → flag
  ↓ Response:
    [Learning Path output đầy đủ]
    "Riêng câu hỏi về lương DA, đây là Phân tích thị trường
     (nhóm D). Mình không xử lý ở nhóm này — bạn ghé nhóm D nhé."
```

### Edge case — Toàn bộ message off-scope ở Mode 1-6

Nếu không có phần nào in-scope → Manager X ask back theo template scope-only:

```
"Nhóm [tên nhóm X] hỗ trợ:
  • [bullet scope nhóm]
Câu hỏi của bạn chưa rơi vào những việc trên — bạn có thể nói cụ thể hơn không?"
```

**KHÔNG cross-suggest group khác** trong template generic này (case-by-case, mỗi Manager tự quyết suggest hay không tuỳ scope agent đích rõ ràng).

## Mismatch entry path vs intent (Mode 1-6)

User click Mode X nhưng message thực tế thuộc Mode Y khác (không compound, chỉ thuần off-scope).

→ Manager X **ask back off-group** (giống edge case ở trên), **không** auto-route, **không** cross-call:

```
User click Mode A → gõ "Tìm việc DA HN 20tr"
  ↓ Manager A nhận
  ↓ Detect toàn bộ off-scope
  ↓ Ask back:
    "Nhóm Khám phá bản thân hỗ trợ MBTI / Holland / Goal setting.
     Câu hỏi của bạn có vẻ về tìm việc — bạn muốn chuyển sang
     nhóm Tìm việc làm không?"
```

User tự navigate (click sang group khác → new conversation theo rule).

## Auto-greet khi vào Mode 1-6

Khi user click card group X từ UI (chưa gõ message gì) → Manager X **auto-greet ngay** với template:

```
"Xin chào! Mình là [tên nhóm X].
 Nhóm này hỗ trợ:
  • [bullet 1]
  • [bullet 2]
  • [bullet 3]
 Bạn muốn bắt đầu với gì?"
```

Auto-greet này **không qua sub-agent** — Manager X tự sinh trực tiếp.

**Mode 0** không auto-greet (user mặc định đã ở free chat khi mở app).

## State filter rule — `handled_by` thống nhất mọi mode

Mỗi message trong `messages` có `metadata.handled_by` ENUM (`supervisor` / `manager_a` / `manager_b` / ... / `manager_f`).

**Nguyên tắc thiết kế thống nhất:** agent KHÔNG biết / KHÔNG quan tâm đang ở Mode 0 hay Mode 1-6. Mọi query metadata đều filter `handled_by` theo agent của chính nó. Cùng 1 code path đúng cho cả 2 mode:

- **Mode 0** (conversation có nhiều Manager đan xen): filter `handled_by` cô lập đúng message của Manager đang query.
- **Mode 1-6** (conversation chỉ có 1 Manager): filter `handled_by` trả full messages (vì tất cả đều thuộc Manager đó) — đúng tự nhiên, không cần case riêng.

### Manager đọc context của mình

```python
def get_my_context(conversation_id, manager: str):
    return SELECT * FROM messages
      WHERE conversation_id = ?
        AND metadata->>'handled_by' IN ('supervisor', manager)
      ORDER BY created_at DESC
      LIMIT N
```

`'supervisor'` được include vì Supervisor là routing layer chung (vd ask-back của Supervisor là context để Manager hiểu vì sao mình được route).

### Memory + Profile — Không filter

2 agent infrastructure này đọc **toàn bộ** recent messages, user-scoped, không quan tâm `handled_by`.

## State filter — Job Search `last_search.filter`

Job Search lưu state ở `messages.metadata.last_search.filter` sau mỗi search (xem [09-job-search.md](./09-job-search.md)).

Khi refine, Manager C scan ngược tìm filter cũ — **luôn filter `handled_by = manager_c`** bất kể mode:

```python
def find_last_search_filter(conversation_id):
    return SELECT metadata->'last_search'->'filter'
      FROM messages
      WHERE conversation_id = ?
        AND role = 'assistant'
        AND metadata->>'handled_by' = 'manager_c'
        AND metadata->'last_search' IS NOT NULL
      ORDER BY created_at DESC
      LIMIT 1
```

- **Mode 0**: filter bỏ qua turn của Manager khác (vd D ở giữa) → tìm đúng filter Manager C gần nhất.
- **Mode 3 (Locked C)**: tất cả assistant message đã có `handled_by = manager_c` → filter trả full → query đúng tự nhiên.

Cùng 1 query, không cần case riêng theo mode.

## Tổng kết — cross-mode behavior

| Aspect | Mode 0 | Mode 1-6 |
|---|---|---|
| Routing | Supervisor classify | Lock vào Manager X |
| Conversation | 1 conversation cho mỗi free chat thread (giống ChatGPT thread) | Mỗi click card = 1 conversation độc lập |
| Switch mode/conversation | New conversation | New conversation |
| Memory + Profile | Mọi mode, mọi turn, user-scoped | Như Mode 0 |
| Compound question | Pick best fit + flag (ii) | In-scope + flag off-scope (i) |
| Off-group thuần | N/A (Supervisor route đúng) | Ask back scope-only |
| Auto-greet | Không | Có, ngay khi user click card |
| State filter (handled_by) | **Unified** — luôn filter `handled_by = manager_x` | **Unified** — cùng query, trả full vì conversation cô lập |
| Job Search refine filter | **Unified** — luôn filter `handled_by = manager_c` | **Unified** — cùng query, trả full |

## Không làm

- KHÔNG đa-Manager song song trong 1 turn (Supervisor pick 1, không gọi 2 Manager).
- KHÔNG auto-route khi user mismatch entry / off-scope — chỉ suggest text, user tự navigate.
- KHÔNG Manager re-route lên Supervisor.
- KHÔNG share conversation_id giữa Mode 0 và Mode 1-6.
